The codebase is well-structured, but several authorization handlers trust client-supplied identity, and live credentials are committed to source control. The Critical section should be read first — and acted on this week.

6Critical 8High 10Medium 6Low
Examiner's summary

The architecture is disciplined — CQRS over MediatR, value objects, a result type instead of exceptions — and most command handlers correctly verify ownership through ICurrentUserContext. The trouble is the handlers that don't: profile updates, post creation, and media uploads all accept a user ID from the request and act on it without checking who is actually calling. Combined with anonymous endpoints that return unpublished drafts, the API's authorization story has real holes.

Separately, secrets hygiene needs immediate attention: an sa database password, an Application Insights connection string, and a working-looking login credential in the Postman collection are all committed to git history. Finally, the password-reset flow is broken end-to-end — the endpoint that is supposed to rescue users who forgot their password requires them to be logged in.

Critical

6 findings — exploitable now, fix immediately

Direct paths to account compromise, data tampering, or credential exposure.

LB-01CriticalSecrets · Source control

Live credentials committed to the repository

src/Lazy.Blog.Api/appsettings.json:3,13appsettings.Development.json:3postman/collections/notlazy.blog.json:64

Three separate secrets are tracked in git: the SQL Server sa password (Password1!) in both appsettings files, a full Application Insights connection string (instrumentation key + ingestion endpoints), and a real-looking login credential (adam.serghei@gmail.com / Password1!) inside the Postman collection.

"Database": "Server=db; Database=Lazy.Blog;User Id=sa;Password=Password1!;TrustServerCertificate=true"
"ApplicationInsights": { "ConnectionString": "InstrumentationKey=33636d32-…" }
RecommendationMove secrets to user-secrets/environment variables/Key Vault, rotate every exposed value (especially if that password is reused for the real account), and scrub git history with git filter-repo. Running the app under sa is itself worth fixing — use a least-privilege login.
LB-02CriticalAuthZ · IDOR

Any authenticated user can overwrite any other user's profile

UpdateUserCommandHandler.cs:16-55UsersController.cs:143-168

PUT /api/users/{id} passes the route ID straight into the handler, which loads and updates that user with no ownership check — unlike its siblings (avatar upload/delete correctly call ICurrentUserContext.IsCurrentUser). Any logged-in account can rename or rewrite any profile, including hijacking usernames.

The controller compounds it with a missing return: even when the handler does fail, the failure is computed, discarded, and 204 NoContent is returned anyway.

if (result.IsFailure)
{
    HandleFailure(result);   // result discarded — falls through
}
return NoContent();
RecommendationInject ICurrentUserContext into UpdateUserCommandHandler and reject when !IsCurrentUser(request.Id); add the missing return in the controller. Also note the handler never checks the UserName/FirstName/LastName.Create results before dereferencing .Value.
LB-03CriticalAuthZ · Impersonation

Posts can be created in any user's name

CreatePostCommandHandler.cs:25-27PostsController.cs:117-145

CreatePostCommand carries request.UserId from the JSON body, and the handler's only check is that the user exists. Any authenticated caller can publish content attributed to any other account — defamation-by-API.

if (await userRepository.GetByIdAsync(request.UserId, ct) is null)  // existence ≠ ownership
    return Result.Failure<PostCreatedResponse>(...);
RecommendationDrop UserId from the request contract entirely and take the author from ICurrentUserContext.GetCurrentUserId(). The same applies to AddCommentRequest.UserId (that handler does verify, but the field is still client-controlled noise).
LB-04CriticalAuthZ · IDOR

Media uploads on behalf of any user

CreateMediaCommandHandler.cs:18-50MediaController.cs:21-33

POST /api/media/{id}/upload uploads into that user's blob folder and records a media item against them, with no check that the caller is that user. The companion query (ListMedia) does check — this one was simply missed. An attacker can fill any user's media library and storage quota.

RecommendationMirror the UploadUserAvatarCommandHandler pattern: fail with UnauthorizedUserUpdate when !currentUserContext.IsCurrentUser(request.UserId).
LB-05CriticalData exposure

Unpublished drafts are exposed to anonymous visitors

PostRepository.cs:64-77GetPostByIdQueryHandler.cs (GetPostByUserId)GetPostBySlugHandler.cs:35-41

Two anonymous read paths leak drafts. GET /api/users/{id}/posts uses GetPostsByUserId, which has no IsPublished filter — draft titles, summaries, and slugs are listed for anyone. With a slug in hand, GET /api/posts/{slug} then returns the full body of the unpublished post, because GetPostBySlugHandler never checks IsPublished either.

IQueryable<Post> posts = dbContext.Set<Post>()
    .Where(p => p.UserId == userId)        // drafts included, endpoint is [AllowAnonymous]
    .OrderByDescending(p => p.PublishedOnUtc)…
RecommendationFilter to published posts unless the requester is the author (the by-username variant already has an includeDrafts concept — reuse it). In GetPostBySlugHandler, return NotFound for unpublished posts unless IsCurrentUser(post.UserId).
LB-06CriticalAuthN · Account takeover

External logins auto-link to existing accounts by email

UserService.cs:94-155AuthController.cs:31-56

In CreateOrLoginUser, when an external sign-in doesn't match a known login, the code looks the user up by email claim and silently calls AddLoginAsync, then issues tokens. There is no check that the provider verified the email (email_verified), no confirmation step, and the email may come from the loose "email" claim fallback. Anyone who can obtain a provider identity asserting a victim's address takes over the account. Google verifies emails today, but the pattern breaks the moment the commented-out Facebook/GitHub/Microsoft providers are enabled.

Related weaknesses in the same flow: FindFirstValue(ClaimTypes.GivenName)! null-forgives claims that Google may omit (NRE → 500), the final user! can genuinely be null, and the callback returns access + refresh tokens as a JSON body on a top-level GET redirect, where they land in browser history and proxy logs. returnUrl is also forwarded unvalidated.

RecommendationOnly auto-link when the provider asserts a verified email, or require the user to sign in with their password once to confirm the link. Deliver tokens via a short-lived one-time code or an HttpOnly cookie rather than a GET response body, and validate returnUrl against an allowlist.

High

8 findings — broken flows & weakened defenses

Not instantly exploitable in isolation, but each one materially weakens the system or breaks a user-facing flow.

LB-07HighAuth flow · Defect

Password reset requires being logged in

ResetPasswordController.cs:12-26ResetPasswordCommandHandler.cs:20-24

ResetPasswordController inherits BaseJwtController ([Authorize] with the JWT scheme) and has no [AllowAnonymous]. The one user who needs /reset-password — someone who can't log in — gets a 401. The forgot-password email is sent, but the link is a dead end.

The handler also throws InvalidOperationException("User not found") for unknown emails: a 500 response that doubles as a user-enumeration oracle, undoing the deliberate anti-enumeration in ForgotPasswordCommandHandler.

RecommendationMake the endpoint anonymous and return the same success-shaped response whether or not the email exists. The reset token already proves authorization.
LB-08HighNetwork trust

Forwarded headers are trusted from any source

Program.cs:41-53

KnownIPNetworks.Clear() + KnownProxies.Clear() tells ASP.NET to accept X-Forwarded-For/Proto/Host from anyone who can reach the app. Clients can spoof their IP (poisoning logs and any future rate-limiting/audit logic), claim HTTPS to neutralize UseHttpsRedirection, and inject a forged host into generated links.

options.KnownIPNetworks.Clear();
options.KnownProxies.Clear();   // = trust every client's X-Forwarded-* headers
RecommendationRegister the actual reverse proxy (KnownProxies.Add(...) or the proxy's network range) instead of clearing both lists. If the proxy IP is dynamic, restrict at least AllowedHosts (currently "*") and pin ForwardLimit to the real hop count.
LB-09HighAuthN · Brute force

No lockout, no rate limiting anywhere

LoginCommandHandler.cs:59Program.cs

Login calls CheckPasswordSignInAsync(user, password, lockoutOnFailure: false), explicitly disabling Identity's account lockout. There is no AddRateLimiter registration in the pipeline, so /api/users/login, /forgot-password (which sends real emails), /api/users/refresh, and the anonymous view-count endpoint are all unthrottled.

RecommendationPass lockoutOnFailure: true and add ASP.NET Core rate limiting with a tight policy on the anonymous auth endpoints.
LB-10HighUpload security

SVG uploads enable stored XSS; size cap is 50 MB

Avatar.cs:9-23

The allowed avatar/media types include .svg — a scriptable format. Files are validated by extension only (no content sniffing) and served from blob storage; an SVG with embedded script becomes stored XSS for anyone who opens the blob URL in a browser context. The size constant is also self-contradictory: 50 * 1024 * 1024 annotated //30 mb, and .png appears twice in the list.

RecommendationDrop SVG (or sanitize/rasterize it), verify magic bytes server-side, lower the cap to something avatar-sized, and serve the container with Content-Disposition/CSP headers that prevent inline rendering as a document.
LB-11HighToken hygiene

Refresh tokens are plain GUIDs, valid six months, stored in plaintext

UserToken.cs:8-21UserTokenRepository.cs:16-18

Guid.NewGuid() is not a cryptographically secure source of randomness (GUID v4 in .NET happens to use a CSPRNG, but the format guarantees only uniqueness, and 6 of its 128 bits are fixed). Tokens live 6 months, are stored and looked up as plaintext (a DB leak = session leak), and rows for used tokens accumulate forever.

RecommendationGenerate with RandomNumberGenerator.GetBytes(32), store only a SHA-256 hash, shorten the lifetime, and purge consumed/expired rows. Consider revoking the family when a used token is replayed (current code detects reuse but leaves all sibling tokens valid).
LB-12HighAuth flow · Defect

External-login refresh tokens are never saved

UserService.cs:152-154JwtProvider.cs:60-64

JwtProvider.GenerateAsync only adds the refresh token to the DbContext; persistence relies on the caller invoking IUnitOfWork.SaveChangesAsync. The login and refresh handlers do — but UserService.CreateOrLoginUser (the Google sign-in path) never saves. The client receives a refresh token that doesn't exist in the database, so the first refresh after an external login fails with UserToken.NotFound.

RecommendationInject IUnitOfWork into UserService and save after token generation — or make GenerateAsync own its persistence so the contract can't be half-fulfilled.
LB-13HighInfo disclosure

Raw exception messages are returned to clients

CustomExceptionHandler.cs:17-27

The global handler copies exception.Message into ProblemDetails.Detail and the exception type name into Type for every unhandled exception — connection strings in SqlException text, file paths, and internal type names all ship to the caller on any 500.

RecommendationLog the exception server-side (the handler currently doesn't log at all) and return a generic detail plus a correlation ID. Reserve specific messages for known, safe exception types.
LB-14HighDependencies

Known-vulnerable package: OpenTelemetry.Api 1.14.0

Lazy.Blog.Api.csproj (transitive via Azure.Monitor.OpenTelemetry.AspNetCore 1.4.0)

The build emits NU1902: OpenTelemetry.Api 1.14.0 has a known moderate-severity advisory (GHSA-g94r-2vxg-569j). Surfaced on every restore; currently ignored.

RecommendationAdd a direct reference to the patched OpenTelemetry.Api (or bump the Azure Monitor distro) and consider failing CI on NU1902+ via <WarningsAsErrors> or dotnet list package --vulnerable in the pipeline.

Medium

10 findings — correctness & operational defects

Bugs that produce wrong results, lose telemetry, or will bite in production under load.

LB-15MediumQuery logic

Pagination filters drafts after taking the page

PostRepository.cs:79-98

In GetPostsByUserName, .Skip(offset).Take(PageSize) runs first and the IsPublished filter is composed afterwards. EF translates this as "take 24 rows including drafts, then discard the drafts" — visitors see short and inconsistent pages, and offsets drift on every page boundary.

.Skip(offset).Take(PageSize);          // page taken here…
if (!includeDrafts)
    posts = posts.Where(p => p.IsPublished);   // …filtered here. Too late.
RecommendationApply the Where before Skip/Take. The OrderByDescending(includeDrafts ? CreatedOnUtc : PublishedOnUtc) conditional also evaluates per-row in SQL rather than choosing a sort — split the query instead.
LB-16MediumMapping bug

"Your vote" is actually the author's vote; null summary crashes listings

PostMapper.cs:24-27,41,46-49

Both list mappers compute the vote badge from p.User.PostVotes — the post author's votes — so every visitor sees whether the author upvoted their own post, never their own vote (the slug endpoint does this correctly via IPostVoteRepository). And p.Summary!.Value null-forgives a nullable value object: one post without a summary turns every feed page into a 500.

RecommendationPass the current user's ID into the query and join votes on it; replace Summary!.Value with Summary?.Value.
LB-17MediumPipeline order

Exception handling is registered at the end of the pipeline

Program.cs:155-178

UseExceptionHandler() and UseStatusCodePages() are added after CORS, authentication, authorization, and endpoint mapping. Exceptions thrown in those earlier middlewares bypass the handler entirely — and failed responses produced there also lack CORS headers, which browsers report as opaque network errors. UseDeveloperExceptionPage() added afterwards never wins either.

RecommendationRegister UseExceptionHandler/UseStatusCodePages first (immediately after UseForwardedHeaders), and gate the developer page with if (env.IsDevelopment()) instead of the production handler, not after it.
LB-18MediumLogging · Telemetry

Log path frozen at startup; App Insights sink writes nowhere

Program.cs:28-34

Three problems in the logger bootstrap: (1) DateTime.Today is captured once, so a process running across midnight keeps writing into yesterday's file forever — the per-day folder scheme only works if the app restarts daily; (2) the path uses Windows backslashes while the project targets Linux containers (DockerDefaultTargetOS=Linux), producing a single oddly-named file; (3) the Serilog sink is built from TelemetryConfiguration.CreateDefault(), which never reads the configured connection string — those events go nowhere, while Azure Monitor OpenTelemetry is registered separately, leaving two half-configured telemetry stacks.

RecommendationUse Serilog's rollingInterval: RollingInterval.Day with forward slashes, and either pass the real connection string to the sink or drop it and rely on the OTel distro's log exporter.
LB-19MediumStartup · Migrations

Migration failures are swallowed and the app starts anyway

Program.cs:153,193-207

CreateDbIfNotExists catches all exceptions, logs, and lets the app serve traffic against a schema that may be missing the latest migrations — turning a clean deploy failure into runtime 500s. Migrating at startup is also unsafe with more than one instance (concurrent migrators race). The TODO acknowledges this.

RecommendationFail fast (rethrow) at minimum; preferably run dotnet ef database update or a migration bundle as a deploy step instead of in-process.
LB-20MediumDead wiring

The auditing interceptor is registered but never attached

Program.cs:88,95-99UpdateAuditableEntitiesInterceptor.cs

UpdateAuditableEntitiesInterceptor is added to DI as a singleton, but AddDbContext never calls optionsBuilder.AddInterceptors(...) — it is dead code. Auditing actually happens manually inside UnitOfWork, which means any save that bypasses the unit of work (everything done through UserManager/SignInManager, e.g. user creation, password resets, external-login linking) silently skips CreatedOnUtc/UpdatedOnUtc.

RecommendationAttach the interceptor in AddDbContext (it has the service provider available) and delete the duplicated logic in UnitOfWork.
LB-21MediumHTTP semantics

Every failure is a 400; not-found and forbidden are indistinguishable

ApiController.cs:14-30PostsController.cs:62-67

HandleFailure maps every non-validation failure to 400 Bad Request — including "post not found" and the unauthorized-ownership errors from LB-02/03/04, which should be 404/403. Clients can't distinguish retryable input errors from authorization denials. (Cosmetic: GetAllPosts checks IsFailure, then re-checks IsSuccess ? Ok : BadRequest on a line that can only be success.)

RecommendationAdd an error-type discriminator to Error (NotFound/Conflict/Forbidden/Validation) and map it to proper status codes in one place.
LB-22MediumResource churn

New Azure/Resend clients constructed per call

FileService.cs:71-79SendGridEmailSender.cs:21

FileService builds a fresh BlobServiceClient and DefaultAzureCredential on every upload/delete — credential chains are expensive (probing IMDS, env, CLI) and are designed to be cached. SendGridEmailSender likewise creates a new Resend client per email. Both classes are also misleadingly named/configured: the "SendGrid" options and class actually drive Resend.

RecommendationRegister BlobServiceClient and IResend as singletons (Resend ships AddResend DI extensions, with HttpClientFactory + retries). Rename the email classes and the "SendGrid" config section to match reality.
LB-23MediumReliability

Registration fails with 500 if the welcome email can't be sent

CreateUserCommandHandler.cs:69-71SendGridEmailSender.cs:13-19

The welcome email is awaited inline after the user is committed; SendEmailAsync throws when the API key is missing/blank (as it is in both committed appsettings). The user is created, but the client receives a 500 and will likely retry into "email already in use". Email dispatch is a side effect that shouldn't gate the response.

RecommendationFire-and-forget with logging, queue it (the domain-events/outbox scaffolding in Idempotence/ is the natural home), or at minimum wrap in try/catch.
LB-24MediumClaims handling

Single() on claims turns odd tokens into 500s

CurrentUserContext.cs:28-34JwtProvider.cs:85-110

IsCurrentUser uses .Single(c => c.Type == NameIdentifier) — a principal with zero (cookie-only/anonymous) or two such claims (easy once Google sign-in adds its own NameIdentifier) throws InvalidOperationException instead of returning false. GetCurrentUserId right below it already does this correctly with SingleOrDefault; the JwtProvider helpers repeat the Single() pattern.

RecommendationUse FirstOrDefault/SingleOrDefault and treat "missing" as not-authorized rather than as an exception.

Low

6 findings — hygiene & debt

Worth fixing opportunistically; none should block a release on its own.

LB-25LowArchitecture

Layer references undercut the clean-architecture intent

Lazy.Blog.Presentation.csprojLazy.Blog.Application.csprojLazy.Blog.Domain.csproj

Presentation references Infrastructure directly (it needs IUserService, which lives in Application — the implementation pull is unnecessary). Application references Microsoft.AspNetCore.Identity.UI (a Razor-pages UI package) apparently just for types that Microsoft.Extensions.Identity.Core would provide. Domain references Identity stores, binding entities to the persistence framework. Infrastructure references Persistence, inverting the usual independence.

RecommendationTrim each project to the minimum reference set; swap Identity.UI for Identity.Core in Application.
LB-26LowTesting · CI

Zero test projects — but the pipeline "runs tests" anyway

azure-pipelines.yml:42-49azure-pipelines-1.yml

The solution contains no test project at all, while the classic pipeline includes a VSTest step globbing for *test*.dll that will match nothing and quietly pass. Two pipelines coexist: a Windows/VSBuild one and a Linux one that only builds (no tests, no publish, no artifact). Auth and ownership logic of this subtlety (see LB-02…LB-07) is exactly what unit tests are for.

RecommendationAdd an xUnit project starting with handler-level tests for the authorization paths; consolidate to the YAML pipeline with build + test + publish.
LB-27LowDead config

Dead configuration and mutable global page size

QueryConstants.cs:5appsettings.Development.json:27

"PostPageSize": 24 in appsettings is never read; the real page size is public static int PageSize = 24 — a mutable global that isn't const/readonly and can't differ per environment. The CancellationToken ct parameters on the IQueryable-returning repository methods are likewise accepted and ignored.

RecommendationBind a typed options class for paging or delete the config key; make the constant const.
LB-28LowBuild warnings

Compiler warnings flag a real null bug and a captured logger

DeleteAvatarCommandHandler.cs:35AuthController.cs:21

CS8604: the avatar URL passed to DeleteByBlobUrl can be null (deleting an avatar twice, or for a user who never set one, throws in new Uri(null)). CS9107: AuthController's primary-constructor logger is captured both in the type and passed to the base. Four warnings total; a codebase this size can hold the line at zero.

RecommendationGuard the null URL (return success if there's nothing to delete), use the base class's logger field, and consider TreatWarningsAsErrors.
LB-29LowEnvironment coupling

Production hostnames hardcoded into startup

Program.cs:112-130SendGridEmailSender.cs:45

CORS origins, the cookie domain (.notlazy.org — which breaks identity cookies on localhost entirely), and the password-reset link base URL are all compiled-in literals. Spinning up a staging environment means editing code.

RecommendationMove origins, cookie domain, and the frontend base URL into configuration with per-environment values.
LB-30LowCode hygiene

Assorted debris: slug hash, GET-with-side-effects, commented-out blocks

CreatePostCommandHandler.cs:36PostsController.cs:193-209UserService.cs:84-92

Slug de-duplication uses postId.ToByteArray().GetHashCode() — the hash of the array reference, not its content, so the "unique" prefix is arbitrary and can be negative. The anonymous PUT …/count-view endpoint lets anyone inflate view counts in a loop (and a view is conceptually a POST). Large commented-out regions (Facebook/GitHub auth, half of CreateExternalUserAsync, role-assignment experiments in LoginCommandHandler), a typo'd response string ("External cooking missing/invalid"), empty <Folder> items in every csproj, and a one-line README round out the list.

RecommendationUse a deterministic suffix (short GUID/base36 of the new post ID) for slugs; gate view counting per IP/session server-side; delete dead code — git remembers it for you.
Recommended order of work
  1. Rotate and purge secrets (LB-01) — assume the committed values are burned.
  2. Close the ownership holes (LB-02, LB-03, LB-04) — three small handler edits plus one missing return.
  3. Stop the draft leak (LB-05) and fix password reset (LB-07) — both are user-visible today.
  4. Re-secure the perimeter: forwarded-header trust (LB-08), lockout + rate limiting (LB-09), exception detail leak (LB-13).
  5. Repair the external-login path (LB-06, LB-12) before enabling more providers.
  6. Then work down the Medium list — most are one-file fixes — and add the first test project (LB-26) so the fixes stay fixed.
